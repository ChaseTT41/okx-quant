### 错题记录 #1

- **问题**：运行 `leverage_engine.py` 的 `fetch_open_positions()` 时报错 `'<' not supported between instances of 'NoneType' and 'str'`
- **我的原回答**：直接使用 ccxt 的 `exchange.fetch_positions()` 获取持仓，未考虑 ccxt 在处理 OKX swap 数据时某些字段（如 market data）为 None 导致内部比较崩溃
- **正确答案**：绕过 ccxt，直接调用 OKX REST API `GET /api/v5/account/positions?instType=SWAP`，用 HMAC-SHA256 签名，手动解析 `instId` 字段（如 `SOL-USDT-SWAP` → `SOL/USDT`），过滤 `pos > 0` 的仓位
- **错误类型**：推理错误
- **错误原因分析**：过度信任 ccxt 库的稳定性。ccxt 是一个"万能适配层"但并非所有交易所的所有接口都经过充分测试。OKX swap 接口返回的某些字段格式与 ccxt 预期不一致（market data 关联查找失败返回 None），导致内部排序/比较时崩溃。我在写代码时没有先验证 ccxt 对 OKX swap 的实际兼容性，直接假设"ccxt 应该能处理"。
- **改进措施**：
  1. 涉及资金/交易的接口，优先使用交易所原生 REST API，ccxt 仅作为备选
  2. 用新 API 前先手动 curl 验证返回结构，确认字段非空
  3. 所有交易所 API 调用都要加 try/except + 明确的日志
- **记录时间**：2026-06-16 13:30

### 错题记录 #3

- **问题**：修改了 `trading_config.py` 中 `max_daily_trades` 默认值从 5→100，但 daemon 日志仍显示"5 笔"
- **我的原回答**：只改了 Python 代码的默认值，认为"改了代码默认值就够了"
- **正确答案**：`.env` 文件中的 `LIVE_MAX_DAILY_TRADES=5` 覆盖了代码默认值。Python 代码用 `os.environ.get("LIVE_MAX_DAILY_TRADES", "100")` 读取，环境变量优先级高于代码默认值。需要同步修改 `.env` 文件。
- **错误类型**：遗漏
- **错误原因分析**：配置加载有"三层覆盖"机制（代码默认 → `.env` → 运行时环境变量），我只修改了最底层。这是对项目配置体系的认知不完整——没有在修改前先理解配置的加载链路。
- **改进措施**：
  1. 改任何配置前，先用 grep 搜索该配置名出现的所有位置
  2. 在心里跑一遍"覆盖链"：环境变量 > .env > 代码默认
  3. 改完配置后必须检查 daemon 日志确认实际生效值
- **记录时间**：2026-06-16 14:00

### 错题记录 #4

- **问题**：Chase哥问"为什么今天美光大涨了你没有提前布局？"
- **我的原回答**：系统没有检测到 MU 的买入信号，没有解释根本原因
- **正确答案**：根本原因是 `strategy_runner.py:42` 中 `ALL_CRYPTO_SYMBOLS = get_all_crypto_symbols(tiers=[1,2])` 只包含 ~80 个加密货币符号，不包含任何 stock perpetual swaps（如 MU-USDT-SWAP）。所以 ML 策略永远看不到 MU。这是系统设计层面的盲区，而非单次信号遗漏。
- **错误类型**：遗漏
- **错误原因分析**：在最初设计策略扫描范围时，ML 策略只覆盖 crypto，stock swaps 没被纳入。这不是单次决策失误，而是架构盲区。更深层的原因是我在设计时没有主动考虑"哪些资产可能大涨但不在当前扫描范围"——缺乏"反向测试"思维。
- **改进措施**：
  1. 新增 `stock_swap_scanner.py` 独立扫描股票合约（已完成）
  2. 定期审查：当前扫描范围是否覆盖了所有可交易资产？
  3. 设计新功能时问自己："什么不在覆盖范围内？"
  4. 对持仓资产（如 MU）设置独立的价格预警，不依赖策略信号
- **记录时间**：2026-06-16 15:00

### 错题记录 #5

- **问题**：`stock_swap_scanner.py` 自测（`if __name__ == "__main__"`）显示 MU 有强烈买入信号，但 daemon 实际扫描结果中 MU 没有信号
- **我的原回答**：自测代码硬编码了 `sector_flow=0.3`（流入），但实际运行时半导体板块的 `flow_score=-0.478`（资金流出），导致信号被正确抑制。自测给了虚假的信心
- **正确答案**：自测应该使用真实的 `MarketSentimentEngine` 数据，而不是硬编码参数。或者至少自测要明确标注"使用模拟参数，仅验证代码逻辑，不代表实际交易信号"
- **错误类型**：事实错误
- **错误原因分析**：写自测时为了方便快速验证代码逻辑，硬编码了乐观参数。但没有意识到这会造成"自测通过但实际不工作"的假象。这是测试设计的基本错误——测试参数不代表真实环境。
- **改进措施**：
  1. 自测代码必须明确区分"逻辑验证测试"和"真实环境测试"
  2. 硬编码参数必须加醒目标注：`# ⚠️ 模拟参数，仅用于逻辑验证，不代表实际信号`
  3. 所有 scanner/strategy 模块增加 `--live-test` 模式，使用真实数据源
  4. 自测输出必须包含数据来源说明（真实 vs 模拟）
