### 错题记录 #1

- **问题**：运行 `leverage_engine.py` 的 `fetch_open_positions()` 时报错 `'<' not supported between instances of 'NoneType' and 'str'`
- **我的原回答**：直接使用 ccxt 的 `exchange.fetch_positions()` 获取持仓，未考虑 ccxt 在处理 OKX swap 数据时某些字段（如 market data）为 None 导致内部比较崩溃
- **正确答案**：绕过 ccxt，直接调用 OKX REST API `GET /api/v5/account/positions?instType=SWAP`，用 HMAC-SHA256 签名，手动解析 `instId` 字段（如 `SOL-USDT-SWAP` → `SOL/USDT`），过滤 `pos > 0` 的仓位
- **错误类型**：推理错误
- **错误原因分析**：过度信任 ccxt 库的稳定性。ccxt 是一个"万能适配层"但并非所有交易所的所有接口都经过充分测试。OKX swap 接口返回的某些字段格式与 ccxt 预期不一致（market data 关联查找失败返回 None），导致内部排序/比较时崩溃。我在写代码时没有先验证 ccxt 对 OKX swap 的实际兼容性，直接假设"ccxt 应该能处理"。
- **改进措施**：
  1. 涉及资金/交易的接口，优先使用交易所原生 REST API，ccxt 仅作为备选
  2. 用新 API 前先手动 curl 验证返回结构，确认字段非空
  3. 所有交易所 API 调用都要加 try/except + 明确的日志

### 错题记录 #10

- **问题**：Chase哥发现 OKX 上有 $0.98 的 ORDI 僵尸仓位和 $3.48 的 ETH 仓位，远低于 $10 最低保证金限制
- **我的原回答**：认为 `calculate_position()` 已算出 $10.48 保证金，MIN_MARGIN 检查已通过，是 OKX 那边的问题
- **正确答案**：`compute_contract_quantity()` 的公式是 `quantity = notional / price`，完全忽略了 OKX 合约的 `ctVal`（合约面值）。对于 ctVal=0.1 的币种（ORDI、ETH 等），每张合约只值 `price * 0.1`，但公式按 `price` 算，导致实际下单量少了 10 倍。
  - ORDI: ctVal=0.1, price=$3.28 → 每张合约值 $0.328 → 下 6 张实际名义价值 $1.96（预期 $20.96）
  - ETH: ctVal=0.1, price=$1741 → 每张合约值 $174 → 下 0.1 张实际名义价值 $17.36（预期 $174）
  - INTC/MU/AVAX: ctVal=1 → 不受影响
- **错误类型**：推理错误
- **错误原因分析**：
  1. `compute_contract_quantity` 的公式来自教科书 `quantity = (margin * leverage) / price`，但这是股票/现货公式，不适用于合约
  2. 没有意识到 OKX 合约有 `ctVal`（合约面值）这个概念——不同币种 ctVal 不同（0.1, 1, 10, 100...）
  3. 从 OKX API 拿到了 ctVal 但没有在计算中使用
  4. 下单后没有验证实际成交是否匹配预期——日志显示"保证金=$10.48"但实际 $0.98，没有告警
- **改进措施**：
  1. `compute_contract_quantity()` 增加 `symbol` 参数 → 从 ccxt market 读取 `contractSize` → 修正公式为 `quantity = notional / (price * contractSize)`（已修，见 `leverage_engine.py:653-676`）
  2. `calculate_position()` 传 `symbol=decision.symbol` 给 `compute_contract_quantity`（已修）
  3. 下单后增加偏差验证: 实际成交 vs 预期名义价值偏差>50% 告警（已修，见 `auto_trade_daemon.py:703-716`）
  4. OKX 上已平掉 ORDI $0.98 和 ETH $3.48 两个僵尸仓位
  5. 所有非 ctVal=1 的币种都需重新验证

## #18 开仓未同步挂SL/TP → 两个仓位裸奔一整晚 — 2026-06-23 03:30

**① 盲从daemon持仓，不做二次评估 (推理错误)**

- 正确做法: 重启后第一件事 = 持仓逐一打分 vs 当前扫描Top3信号，弱者直接砍

**② 被动心态: "等止盈"而非"找最优" (遗漏)**
- 扫描结果明确显示MRVL(90分)>NVDA(87分)>ASML(87分)远优于META(弱)和AMD(近TP)

- 这是典型的被动等待思维——等市场替我决策，而不是我主动决策
- Chase哥纠正: 有利润+有更好的标的 = 立即调仓，不要等
- 正确做法: 看到90分信号 → 立刻算"平哪个弱仓能腾出保证金" → 执行

**③ 小账户调仓摩擦被忽略 (遗漏)**
- 平2仓+开2仓 = 4次spread × ~0.15% ≈ $0.57 = 账户的1.2%
- 在$47小账户上，每次调仓的摩擦成本是巨大的
- 我没有在调仓前计算"摩擦成本 vs 预期收益提升"
- 正确做法: 调仓前估算 `friction_cost / equity`，如果>0.5%需要更高把握

| 隔夜持仓自然波动 | -$1.87 | 3.8% |
